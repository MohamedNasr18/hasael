## 🧠 Project context

You are continuing the development of **HASAEL** — an integrated digital agricultural ecosystem built with **Node.js + Express + MongoDB (Mongoose)**.

HASAEL is structured around **2 active marketplaces**, all accessible through a single **Unified Account System**:

- **Marketplace 1 — Agricultural Investment & Operations**: connects investors with farm owners. Farm owners list their lands and publish investment opportunities; investors browse the feed, submit interest or investment requests, and manage deals.
- **Marketplace 2 — Agricultural Services**: connects farm owners and investors with service providers (machinery, irrigation, logistics, consultancy). *(backend scaffolding required — see section below)*

**Unified Account System**: one user account can hold multiple roles simultaneously (`farm_owner`, `investor`, `service_provider`). The `role` field on the User model is an array. A user switches context (active marketplace + active role) via a `active_role` field on their session or profile — not by creating separate accounts. All resources (farms, investments, services) are always scoped to `req.user._id` regardless of which marketplace the request comes from.

## 📐 Architecture rules — follow strictly

1. **Every controller function must be wrapped with `asyncHandler`** — no try/catch blocks.
2. **Every route must go through the `protect` middleware** unless it is a public auth route (login/register).
3. **Validation with Joi** — every POST and PATCH route must have a Joi validator middleware before the controller.
4. **Responses follow this exact shape:**
```json
{
  "success": true | false,
  "message": "...",
  "data": { } | [ ] | null,
  "pagination": { } // only on list endpoints
}
```
5. **Ownership guard pattern** — any resource that belongs to a user must check `resource.owner.toString() === req.user._id.toString()` before update or delete.
6. **Status guard** — resources with a `status` field must block edits when status is terminal (e.g. `funded`, `closed`).
7. **Whitelist updates** — never use `Model.findByIdAndUpdate(id, req.body)` directly. Always whitelist allowed fields explicitly.
8. **Nested object updates** — use spread merge for nested fields (e.g. `investment_terms`, `location`) so partial updates don't wipe existing data.

## 🗄️ Existing data models

### User — Unified Account
- `full_name`, `email`, `password` (select: false), `phone`, `avatar`
- `roles`: `[String]` — array, enum: `farm_owner | investor | service_provider`. One account can hold multiple roles at the same time.
- `active_role`: `String` — the role currently active (must exist in `roles[]`). Scopes which marketplace context and feed the user sees.

- Add `PATCH /api/users/me/switch-role` — body: `{ role }` — switches `active_role`. Guard: role must already exist in `roles[]`.

- `is_published` (bool), `published_at` (auto-set on first publish via pre-save hook)
- `status`: available | under_negotiation | funded | closed

## ✅ What needs to be built

### 1. Investment model + CRUD
Create `models/investmentSchema.js`:
- `farm` → ref: Farm
- `investor` → ref: User
- `owner` → ref: User (farm owner — for quick ownership checks)
- `amount` (Number, optional)
- `message` (String)
- `status`: pending | accepted | rejected | cancelled
- `type`: express_interest | full_investment
- `timestamps: true`

Controllers needed (`controllers/investmentController.js`):
- `createInvestment` — POST /api/investments — investor submits interest or investment request
  - Guard: investor cannot invest in their own farm
  - Guard: farm must be `available` and `is_published: true`
  - On create: increment `farm.interest_count` by 1
- `getMyInvestments` — GET /api/investments/my — investor sees all their requests (paginated, filterable by status)
- `getFarmInvestments` — GET /api/investments/farm/:farmId — farm owner sees all requests on their farm
  - Guard: only the farm owner can access this
- `respondToInvestment` — PATCH /api/investments/:id/respond — farm owner accepts or rejects
  - Guard: only farm owner can respond
  - On accept: update farm status to `under_negotiation`
  - On reject: no farm status change
- `cancelInvestment` — PATCH /api/investments/:id/cancel — investor cancels their own pending request
  - Guard: only the investor who created it, and only if status is `pending`

### 2. Feed endpoint
Create `controllers/feedController.js`:
- `getFeed` — GET /api/feed
  - Returns all farms where `is_published: true` AND `status: available`
  - Paginated (cursor-based or offset — your choice, be consistent)
  - Filterable by: `governorate`, `crop_type`, `min_size_feddan`, `max_size_feddan`
  - Sorted by `published_at` descending
  - Each farm card returns: `name`, `location`, `crop_type`, `size_feddan`, `investment_terms`, `cover_image`, `owner` (name only), `interest_count`
  - On each fetch: increment `views_count` for each returned farm (use `bulkWrite` for performance)

### 5. Error handler middleware
Create `middleware/errorHandler.js` and mount it at the end of `app.js`:
```js
app.use(errorHandler);
```
It must handle:
- Mongoose `ValidationError` → 400 with field-level messages
- Mongoose `CastError` (invalid ObjectId) → 400 "Invalid ID format"
- Mongoose duplicate key (`code 11000`) → 409 with the duplicate field name
- JWT errors (`JsonWebTokenError`, `TokenExpiredError`) → 401
- Generic errors → 500

Shape:
```json
{
  "success": false,
  "message": "Human-readable error",
  "errors": { } // field-level, only for ValidationError
}
```
